# Architecture & Security Model

## Why S-Flag Mock-TCP is the Default

During high-payload production testing (specifically under active TLS handshake data), the legacy mock-TCP `"PA"` (Push+Ack) flags were blacklisted/dropped by stateful network firewalls and ISP middleboxes. This payload-layer blackholing resulted in HTTPS handshake timeouts.

Switching to the `"S"` (SYN) flag simulates standard TCP connection setup flags. This bypasses stateful inspection of active payload streams. Thus, `"S"` flags are the validated production default for both client-side and server-side endpoints.

## Raw Socket (AF_PACKET) and Kernel Suppression

Because the tunnel uses raw `AF_PACKET` sockets to craft and write custom TCP packets bypass-passing the standard OS kernel TCP/IP stack, the local kernel is unaware of the active tunnel socket states.

```
       +------------------------------------+
       |          Paqet Application         |
       |  (Crafts custom TCP SYN packets)   |
       +------------------------------------+
            |                          ^
      (Raw AF_PACKET)             (Raw AF_PACKET)
            |                          |
            v                          |
      +------------+             +------------+
      |  Outgoing  |             |  Incoming  |
      |   Packet   |             |   Packet   |
      +------------+             +------------+
            |                          |
            |   +------------------+   |
            +-->|   Local Kernel   |<--+
                | (Connection is   |
                |  untracked: sends|
                |  RST to reject)  |
                +------------------+
```

When the remote endpoint replies with an incoming packet (e.g. a SYN-ACK), the local kernel TCP stack sees a packet for an untracked connection. By default, the kernel replies with a `RST` (Reset) packet to reject the "invalid" traffic. If this `RST` packet reaches the wire, it immediately terminates the tunnel stream.

### Suppressing RSTs via iptables

To prevent the local kernel from tearing down the tunnel:
1. **Raw NOTRACK rules**: Exempt the transport port traffic from connection tracking (`NOTRACK` in `raw` table). This prevents the CPU from wasting overhead on connection state tracking.
2. **Mangle RST drop rules**: Drop outgoing RST packets generated by the kernel for this port/IP, while letting the custom raw socket packets pass.

These rules must be configured **on both the client side and the server side** for raw socket mock-TCP to work.

## Systemd Ordering & Boot Persistence

For reboot durability, the iptables rules must be applied **before** the Paqet server or client processes start. If the tunnel service starts before the iptables rules are active, the kernel will send RSTs and the tunnel will fail to connect.

This is achieved using a systemd oneshot helper service (`<service>-iptables.service`) which runs the idempotent rules script. A systemd drop-in configuration (`10-iptables.conf`) forces the tunnel service to wait for the iptables service:

```ini
[Unit]
Wants=recoba-paqet-tunnel-iptables.service
After=recoba-paqet-tunnel-iptables.service
```
